## 1. SQL: первичные ключи

### Создание таблицы с PRIMARY KEY + автоинкрементом

`IDENTITY(seed, increment)` — автогенерация значения: `IDENTITY(1,1)` начинает с 1 и увеличивает на 1.

```sql
CREATE TABLE Category (
    CategoryID  INT IDENTITY(1,1) NOT NULL,
    Name        NVARCHAR(100)     NOT NULL,
    CONSTRAINT PK_Category PRIMARY KEY (CategoryID)
);
```

Короткий inline-вариант (без явного имени constraint — имя сгенерит сервер):

```sql
CREATE TABLE Product (
    ProductID   INT IDENTITY(1,1) PRIMARY KEY,
    Name        NVARCHAR(200)  NOT NULL,
    Price       DECIMAL(10,2)  NOT NULL,
    CategoryID  INT            NULL
);
```

> Именованный constraint (`PK_Category`) лучше: его проще найти, изменить и удалить позже.

### Первичный ключ одной строкой

Когда PK — это одно автоинкрементное поле, всё описание умещается в одну строку прямо в определении столбца:

```sql
UserID INT IDENTITY(1,1) PRIMARY KEY
```

Например, таблица пользователей:

```sql
CREATE TABLE [User] (
    UserID    INT IDENTITY(1,1) PRIMARY KEY,
    Login     NVARCHAR(50)  NOT NULL,
    Password  NVARCHAR(100) NOT NULL,
    RoleID    INT           NOT NULL
);
```

> `User` — зарезервированное слово в T-SQL, поэтому имя таблицы берём в квадратные скобки: `[User]`.

### Добавить PK к уже существующей таблице

```sql
-- столбец сначала должен быть NOT NULL
ALTER TABLE Product ALTER COLUMN ProductID INT NOT NULL;

ALTER TABLE Product
ADD CONSTRAINT PK_Product PRIMARY KEY (ProductID);
```

### Составной первичный ключ

Когда строку уникально определяет комбинация полей (типичный случай — таблица-связка многие-ко-многим):

```sql
CREATE TABLE OrderItem (
    OrderID    INT NOT NULL,
    ProductID  INT NOT NULL,
    Quantity   INT NOT NULL,
    CONSTRAINT PK_OrderItem PRIMARY KEY (OrderID, ProductID)
);
```

### Внешний ключ (FK на PK другой таблицы)

```sql
ALTER TABLE Product
ADD CONSTRAINT FK_Product_Category
FOREIGN KEY (CategoryID) REFERENCES Category(CategoryID);
```

### Удалить PK

```sql
ALTER TABLE Product DROP CONSTRAINT PK_Product;
```

### Когда ключ НЕ автоинкремент

`IDENTITY` работает только с числовыми типами (`int`, `bigint`). Если первичный ключ строковый (`nvarchar`) — например, артикул товара — автоинкремента не будет, следующий ID придётся вычислять вручную в коде:

```csharp
// max+1 для строкового ключа: парсим в число, берём максимум, +1
private string GetNextId()
{
    var ids = db.Items.Select(i => i.ItemId).ToList()
        .Select(s => int.TryParse(s, out int n) ? n : 0);
    return ((ids.Any() ? ids.Max() : 0) + 1).ToString();
}
```

> Для числового ключа проще: `db.Orders.Max(o => o.OrderId) + 1` прямо в запросе. Строковый ключ нельзя сравнивать как число напрямую (по алфавиту "9" > "10"), поэтому сначала `ToList()`, потом парсинг.

### Почему это важно для EF

EF Database First строит сущности, связи и навигационные свойства **по PK и FK**. Если у таблицы нет первичного ключа, EF сделает её read-only (или не замапит вообще). Правило простое: **у каждой таблицы должен быть PK**, а связи между таблицами — через FK.

## 2. Генерация SQL-скрипта средствами SSMS

### Вся база целиком (схема и/или данные)

1. ПКМ по базе → **Tasks** → **Generate Scripts…**
2. **Choose Objects** — вся база или выбранные таблицы.
3. **Set Scripting Options** → кнопка **Advanced**.
4. Параметр **Types of data to script**:
   - `Schema only` — только структура (CREATE TABLE и т.д.);
   - `Data only` — только INSERT-ы с данными;
   - `Schema and data` — структура + данные.
5. Output: в файл, в буфер обмена или в новое окно запроса → **Next** → **Finish**.

### Одна таблица (CREATE-скрипт)

ПКМ по таблице → **Script Table as** → **CREATE To** → **New Query Editor Window**.

### Только данные одной таблицы (INSERT-ы)

Через **Generate Scripts** → выбрать таблицу → Advanced → **Data only**.

> Для сдачи проекта удобно держать в репозитории один полный скрипт `Schema and data`, чтобы базу можно было развернуть с нуля на любой машине.

## 3. Сетап проекта с нуля

### Тип проекта

**Windows Forms App (.NET Framework)**, язык C#, версия **4.7.2** или **4.8**.

> Важно: именно **.NET Framework**, а не .NET Core / .NET 5+. Визуальный дизайнер EDMX и EF6 Database First работают только на Framework.

### NuGet-пакеты

Нужен один пакет:

| Пакет | Назначение |
|-------|------------|
| `EntityFramework` (6.x) | ORM, провайдер SQL Server идёт в комплекте |

Установка через **Package Manager Console**:

```powershell
Install-Package EntityFramework
```

Или: ПКМ по проекту → **Manage NuGet Packages** → Browse → `EntityFramework` → Install.

### Добавление Entity-модели (Database First, EDMX)

1. ПКМ по проекту → **Add** → **New Item…**
2. Раздел **Data** → **ADO.NET Entity Data Model** → имя (напр. `AppDbModel`) → **Add**.
3. **EF Designer from database** → **Next**.
4. **New Connection…** → провайдер **Microsoft SQL Server**:
   - Server name: `localhost`, `.\SQLEXPRESS` или имя твоего инстанса;
   - Authentication: Windows / SQL Server;
   - выбрать базу из списка → **OK**.
5. Поставить галочку «**Save connection settings in App.config**», задать имя контекста (напр. `TradeBaseEntities`).
6. Выбрать версию **Entity Framework 6.x**.
7. Отметить нужные **Tables / Views / Stored Procedures** → **Finish**.

Результат: файл `.edmx`, класс `DbContext`, классы сущностей и connection string в `App.config`.

### Использование в коде

```csharp
using (var db = new TradeBaseEntities())
{
    var products = db.Product
        .Where(p => p.Price > 0)
        .OrderBy(p => p.Name)
        .ToList();
}
```

### Если изменилась схема БД

Открыть `.edmx` → ПКМ по пустому месту дизайнера → **Update Model from Database** → вкладки **Add / Refresh / Delete**.

## 4. Что загрузить в репозиторий (КОД 09.02.07-2-2026)

> Перечень взят из задания, раздел «Практические результаты» (Модуль 4) и из требований к артефактам по модулям. Текст задания первичен — при расхождении ориентируйся на него.

Дословно из задания, практические результаты:

- исходный код приложения (**структура с файлами, не архив**);
- **исполняемые файлы**;
- файл скрипта базы данных;
- прочие графические/текстовые файлы.

Плюс артефакты-документы, разбросанные по модулям (см. раздел 5).

Примерная структура:

```
exam-repo/
├── src/                    ← исходный код: .sln + проект, структура с файлами (НЕ архив)
├── bin/                    ← исполняемые файлы (.exe и зависимости) — ТРЕБУЕТ задание
├── database/
│   └── script.sql          ← скрипт БД (схема + данные) или .dt для 1С
└── docs/
    ├── er-diagram.pdf       ← ER-диаграмма в PDF (таблицы, связи, атрибуты, ключи)
    ├── algorithm.pdf        ← блок-схема алгоритма (ГОСТ 19.701-90) в PDF
    └── report.docx          ← скриншоты корректной работы приложения
```

### Ключевые отличия от «обычного» репозитория

На ДЭ есть два пункта, которые противоречат привычным правилам Git — не перепутай:

- **Исполняемые файлы коммитим.** В обычных проектах `bin/` не кладут в репозиторий, но задание прямо требует «исполняемые файлы». Так что `bin/Debug` с `.exe` загружаем. Это исключение именно для ДЭ.
- **Диаграммы в PDF, а не PNG.** Задание дважды указывает формат **PDF** для ER-диаграммы и для блок-схемы. Картинку (`.png`) не примут как замену — экспортируй/печатай в PDF.
- **Диаграмма классов НЕ требуется.** В этом КОД её нет в требованиях — не трать на неё время.

### README (по желанию, но полезно)

Короткий файл с порядком развёртывания: создать БД из скрипта → проверить connection string в `App.config` → собрать решение (`Build Solution`) → запустить. Помогает эксперту быстро поднять проект.

## 5. Артефакты-документы по модулям (что и где требуется)

Помимо кода, задание требует отдельные документы. Их легко забыть, а это прямые баллы. Сводка по модулям:

| Документ | Формат | Модуль | Требование |
|----------|--------|--------|-----------|
| Скрипт БД (схема + данные) | `.sql` / `.dt` | 1 | Сохранить результаты создания БД |
| ER-диаграмма | **PDF** | 1 | Таблицы, связи, **атрибуты и ключи** |
| Блок-схема алгоритма | **PDF** | 2 | По ГОСТ 19.701-90 |
| Отчёт со скриншотами | `.docx` | 2 и 3 | Скриншоты корректной работы системы |

### ER-диаграмма — как сделать в PDF

Самый быстрый путь через SSMS: ПКМ по узлу **Database Diagrams** → **New Database Diagram** → добавить все таблицы → сохранить. Затем экспортировать в PDF (через печать в «Microsoft Print to PDF» или сторонний инструмент). Важно: задание требует, чтобы на диаграмме были видны **атрибуты и ключи**, а не только прямоугольники таблиц со связями — проверь, что столбцы и пометки PK/FK отображаются.

### Блок-схема алгоритма — ГОСТ 19.701-90

Оформляется в любом ПО для диаграмм (на площадке есть «ПО для построения и редактирования диаграмм UML и блок-схем»). Основные элементы ГОСТ: овал — начало/конец, прямоугольник — действие, ромб — условие, параллелограмм — ввод/вывод. Экспорт в PDF.

### Отчёт docx

Документ Word со скриншотами работающего приложения: список товаров с подсветкой, фильтрация, формы добавления/редактирования, заказы. Цель — показать, что приложение запускается и работает без аварийного завершения.

## 6. Типовые грабли (проверено на практике)

Места, где на ДЭ застревают чаще всего. Все — из реальной отладки этого проекта.

### Грязные данные: пробелы, ё/е, регистр

Данные на ДЭ намеренно «грязные»: роль `Авторизованный` (через «о», легко спутать с «и»), статус `Новый ` с висячим пробелом, `Завершен` без «ё». При сравнении строки из базы с захардкоженной (`switch` по ролям, выбор статуса в combobox) одна несовпадающая буква или пробел — и ветка молча не срабатывает.

Правило: везде, где сравниваешь строку из БД, добавляй `?.Trim()`, а лучше грузи значения **из самой базы** (`Distinct().ToList()`), а не пиши руками.

### Привязка к DbSet напрямую

`dataGridView.DataSource = db.Items` — ошибка. Нужно `db.Items.Local.ToBindingList()` (для редактирования) или `db.Items.ToList()` (для просмотра). DbSet — это запрос, а не данные.

### Два контекста не видят правок друг друга

Если форма списка и форма редактирования имеют свои `db`, после сохранения список не обновится: его контекст держит старый кэш. Решение — пересоздать контекст при перечитывании:

```csharp
private void reloadAll()
{
    db.Dispose();
    db = new DEMOEKZ_TESTEntities();   // новый — без кэша
    allItems = db.Items.ToList();
    applyFilters();   // ВАЖНО: перерисовка в конце, иначе данные обновятся, а карточки нет
}
```

Порядок важен: сначала пересоздать контекст и прочитать, **потом** рисовать. Если перерисовать на старом контексте, а обновить после — карточки покажут старое до следующего фильтра.

### Image.FromFile лочит файл

`pictureBox.Image = Image.FromFile(path)` держит файл открытым, пока картинка висит в контроле. Потом этот файл нельзя удалить/заменить — «процесс не может получить доступ». Грузи через копию в память:

```csharp
using (var temp = Image.FromFile(path))
    pictureBox.Image = new Bitmap(temp);   // копия, файл сразу освобождается
```

Применять **везде**, где грузишь картинку (и в карточке, и в форме) — иначе один залоченный файл всплывёт неожиданно.

### OutOfMemoryException на картинке

`Image.FromFile` кидает «недостаточно памяти» не из-за памяти, а когда файл **битый или не картинка** (0 байт, переименованный). Оборачивай в try/catch, чтобы один плохой файл не ронял всё приложение.

### NumericUpDown обрезает по Maximum

По умолчанию `Maximum = 100`. Если присвоить `Value = 5000` без поднятия потолка — молча обрежет до 100, и сохранишь испорченное число. Для цены ставь большой `Maximum` и `DecimalPlaces = 2` для копеек. Бонус: `Minimum = 0` автоматически запрещает отрицательные значения (требование спеки).

### MouseClick не всплывает от дочерних контролов

Клик по лейблу/картинке внутри `UserControl` обрабатывает сам лейбл, а не карточка. Чтобы клик ловился по всей карточке — подписывай и вложенные контролы:

```csharp
card.MouseClick += (s, e) => CardClick(item, e);
foreach (Control c in card.Controls)
    c.MouseClick += (s, e) => CardClick(item, e);
```

### Удаление с внешним ключом

Товар, на который ссылается заказ (`OrderItems`), база не даст удалить — сработает FK. Это не баг, а защита целостности. Оборачивай `SaveChanges()` в try/catch и показывай понятное сообщение. У `EntityException` реальная причина обычно в `InnerException`, а не в верхнем сообщении.

### Конфликт имён Application / Image

`using static System.Net.Mime.MediaTypeNames;` тащит в файл `Application` и `Image`, конфликтующие с `System.Windows.Forms.Application` и `System.Drawing.Image`. Удали лишний `using` — VS подсвечивает неиспользуемые серым.

### Если правки «не применяются»

Прежде чем искать баг — пересобери (`Ctrl+Shift+B`). VS иногда запускает старую сборку, и кажется, что код не работает.
